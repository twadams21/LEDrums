# S22 — Colour batch 4: particles (Group F, issue #51)

## Summary

Colour batch 4 applies the S19 recipe to the ten particle/spatial effects, closing the
professional-show colour pass over all 41 registry effects. Every particle effect now exposes a
real `saturation` threaded into every `hsvToRgb` call, so saturation 0 desaturates a lit effect to
white and pastels become possible where before the S-slot was hardcoded.

Six effects carry a genuine base `hue` and light up the write-through **ColorSwatch** automatically
(no per-effect UI): **starfield** (default saturation **0.15** — its old near-white tint),
**comet-trails** (base hue + per-comet ±30° offset, the `synced-hoops` shape), **lightning**
(default saturation **0.35** — its old blue-white core), **helix** and **orbit-rings** (base hue +
spatial spread), and **collisions** (base node `hue` drives the swatch; `flashHue` stays a separate
accent, both desaturated by the shared saturation).

Four are intrinsically multi-colour and stay swatch-less, taking a range/offset scheme + shared
saturation (the S20 `temp-sweep` precedent): **confetti-burst** and **gravity-wells** remap their
random full-spectrum hues to `baseHue`+`hueSpan` (defaults 0/360 reproduce the old `rng()*360`
draw byte-for-byte); **velocity-flames** exposes its hardcoded yellow→red palette as
`baseHue`/`tipHue` endpoints with `saturation` scaling the intrinsic whiter-base/redder-tip ramp;
**sacred-hogs** already had `hogHue`+`haloHue` and just gains the shared saturation.

Defaults reproduce the pre-S22 output exactly. Core-only + tests + audit table; no `ParamValue`
widening (S18), no `mapParamSpec` / ColorSwatch / design-system change.

## Context pack (brief — the colour pass is now complete)

- **Recipe unchanged from S19/S20.** Add `saturation` after `hue` (default = the effect's old
  hardcoded S-slot: 1, or 0.15 for starfield / 0.35 for lightning), read `pnum(params,'saturation',
  default)`, swap every `hsvToRgb(hue, <const>, …)` → `hsvToRgb(hue, sat, …)`. Swatch is automatic
  once hue+saturation+brightness all exist.
- **Multi rule (as applied here).** An effect with no single meaningful base hue is *multi* → no
  bare `hue` key, so no swatch. Two kinds appeared in this batch: random-palette effects
  (confetti-burst, gravity-wells — each particle/well a random hue) got a `baseHue`+`hueSpan` range
  that constrains the palette; a hardcoded-gradient effect (velocity-flames) got its palette
  endpoints (`baseHue`/`tipHue`) exposed. All are byte-identical at defaults (verified by the
  parity test). `collisions` and `sacred-hogs` already had two hue params; only saturation was
  added — `collisions` keeps a swatch (its `hue` is the primary), `sacred-hogs` does not (hog+halo
  are co-equal accents).
- **`confetti-burst` golden uses a peak-lit-frame helper** (`sweepPeak`): confetti spawns at the
  drum origin (~100 mm from the nearest pixel, past the ~40 mm `reach` floor) and flies outward, so
  it only lights the pixel shell *mid-flight* — the last swept frame is empty. Deterministic given
  the seeded state + fixed frame schedule.
- **Audit tally after S22:** batch 1 (9) + batch 2 (9) + batch 3 (13, S21) + batch 4 (10) = **41 of
  41** — every registry effect accounted for.
