# S02 — OutputPill truth from OutputStatus (B, issue #47)

## Summary
The engine-link pill now tells the truth. It derives from the server's
`OutputStatus` (arming state, `packetsSent`, `lastError`, `universeCount`) — already
broadcast on `state` at connect and in every 500ms `stats` tick — combined with the
WS link state, instead of link state alone. The pill shows distinct, honest states:
**LIVE** (armed + packets, no error), **ERR** (armed but erroring — red, pulsing, the
actual error in the tooltip), **DRY** (dry-run), **ARMED** (armed but no packets yet),
**OFF** (disabled), **SYNC** (connecting), **LOCAL** (offline). This kills the worst
lie in the app: a green "LIVE" while Art-Net was failing (`lastError` set, packets 0).

## Context pack (for S03 — output status panel)
S03 (Patch controller inspector status section) consumes the **same stream**:
- **Derivation seam:** `apps/web/src/lib/app/chrome/output-pill.ts` — `deriveOutputPill(link, output)`
  is a pure `(LinkState, OutputStatus | null) → { tone, label, title, pulse }`. Reuse the
  same tone vocabulary (`live`/`warn`/`muted`, red = LIVE/error, amber = dry-run/warn) and the
  same "error prominently styled" convention for panel consistency.
- **Store field:** `TriggerLab.output` (`store.svelte.ts:~276`) is the adopted latest
  `OutputStatus`, refreshed in both `onState` (connect) and `onStats` (500ms). null until the
  first message. Read this directly for the panel; no new plumbing needed.
- **packets/s note:** S02 intentionally uses cumulative `packetsSent > 0` as the snapshot-level
  "has transmitted" signal — the pure function takes a single snapshot and cannot compute a rate.
  **True packets/s (armed but stalled) is S03's job**: diff `packetsSent` across successive `stats`
  ticks (a stateful derivation, unit-testable per the slice). When S03 lands, consider feeding a
  computed "flowing" boolean into a future pill refinement, but that is out of S02 scope.
- `OutputStatus` wire shape: `packages/protocol/src/index.ts:128` (re-exported to web via
  `apps/web/src/lib/ws/protocol-types.ts`).

## Acceptance
- [x] **Distinct states — live+transmitting, armed-but-erroring (error tone + last error in
      tooltip), dry-run, disabled, link-closed.** `deriveOutputPill` returns a distinct
      `(tone, label, pulse, title)` per state; ERR reuses the red `live` tone but is set apart
      by a pulsing dot + "ERR" label + `Output error: <lastError>` tooltip. Also surfaces an
      honest **ARMED** (armed, packets 0) so the "not yet flowing" case isn't mislabeled LIVE.
      (`output-pill.ts`, styleguide demo renders all seven.)
- [x] **Table-driven unit tests over the pure derivation.** `output-pill.test.ts` — 14 tests: a
      10-row state table plus universe-count singularisation and three invariant guards.
- [x] **"LIVE" impossible while `lastError` set or packets aren't flowing.** Enforced by branch
      order (link ≠ open → not live; `lastError` → ERR; armed && `packetsSent===0` → ARMED — all
      before LIVE) and locked by three dedicated invariant tests: never LIVE with `lastError` set
      (any state), never LIVE with `packetsSent===0`, never LIVE while link not open.

  pulse + label + tooltip. Noted for the design system if a first-class `danger` tone is wanted later.
