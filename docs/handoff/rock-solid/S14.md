# S14 — Drum-link indicator + unrouted-input event (E, issue #50)

## Summary
Both-fire is kept by design (doc 03 §4, LOCKED) — a MIDI/OSC message that is BOTH zone-mapped
to a drum AND bound to an authored graph fires both paths per hit. S14 makes that **visible**
instead of removing it, and surfaces genuinely **unrouted** input the Monitor previously
swallowed:

- A pure resolver `zoneLinkForSource(inputMap, source)` decides whether a trigger source is
  ALSO a mapped drum zone; the trigger node face shows a small **link badge** whose hover
  tooltip names the drum · zone (`"also drum trigger: kick · center"`). The **same hint**
  shows in the `TriggerSourceInspector` (forward: this source also fires a drum) and, in the
  reverse direction, in the `PatchZoneInspector` (`"also fires: <graph names>"`).
- The core engine emits a new **`input-unrouted`** diagnostic when a raw MIDI note / OSC
  address matches **nothing** (no zone-map pad AND no direct graph source). The server host
  maps it to a distinct **"Unrouted input"** Monitor event — kept apart from `graph-missed`,
  which is a *routed* drum hit whose active section simply holds no graph.

No engine resolution behaviour changed (both-fire intact); the icon/hints are display-only,
and `input-unrouted` only re-labels the previously-conflated `graph-missed/no-direct-match`
case for a raw unmatched message.

## Context pack (brief)

**The pure seam** is `apps/web/src/lib/app/trigger-source-label.ts` (beside
`describeTriggerSource`), three pure functions, all unit-tested:
- `zoneLinkForSource(inputMap, source) → {drumId, zone} | null` — a `midi{note}` matched in
  `inputMap.midiNotes`, or an `osc{address}` matched in `inputMap.oscMap`, returns that zone
  (`zone` is the numeric slot string / padKey form, so `describeTriggerSource` renders it).
  Null for unbound / `drum` / `cc` (the zone-map keys notes, not CCs) / unmatched sources.
- `drumLinkHint(inputMap, source, drums) → string | null` — the ONE phrasing
  (`"also drum trigger: kick · center"`) the node badge tooltip AND the source inspector share.
- `graphsLinkedToZone(graphs, note, address) → string[]` — the reverse: authored graph keys
  whose trigger source is that note/address (for the zone inspector; caller maps keys → labels).

**The UI reads the authoritative patch input map** (`store.project.inputMap`) — so the badge/
hints only appear when a project is present (server-authoritative; null offline, like the rest
of the zone inspector). The badge rides `NodeCard`'s new optional `badge` snippet slot (a
top-right corner marker; play-node thumbs sit on the right, trigger nodes never have thumbs, so
there's no collision).

**The diagnostic split** (core `engine.ts` `processEvent`, on `toFire.length === 0`):
`(noteOn|osc) && !drumId` → `input-unrouted` (the server zone-map didn't claim it AND no direct
graph matched — genuinely unbound); everything else (a routed drum hit / `key` with no graph)
stays `graph-missed`. Host `voice-engine-host.ts monitorVoiceDiagnostic` maps the new kind to a
`{type:'graph', label:'Unrouted input'}` Monitor draft (placed before the `section-recalled`
fallback so the union narrows).

## Deviations
- **`input-unrouted` replaces (not adds to) the old `graph-missed/no-direct-match` label for a
  raw unmatched message.** Doc 03 says misses were "engine-side only"; in fact issue #42 already
  surfaced `graph-missed` to the Monitor, so a second event per miss would be noise. Instead the
  raw-unmatched case is re-labelled to the more specific `input-unrouted` (distinct Monitor line),
  and `graph-missed` is reserved for a routed hit with no graph. The one core test and one host
  test that asserted the old label for a raw note were updated to the new contract (in scope —
  they test exactly the behaviour S14 changes); a new host test keeps `graph-missed` covered.
- **Badge lives on a new `NodeCard.badge` slot, not a pre-existing "icon-chip slot".** Doc 03
  described rendering "via NodeCard's icon-chip slot" but no such slot existed; adding a small,
  reusable corner `badge` snippet is the cleanest design-system extension (demoed in the
  styleguide + regenerated in the same change).
- **No DOM-level test of the node badge / inspector rows.** The acceptance criterion scopes the
  "exactly when" evidence to *pure helper* unit tests, which are complete; the components are
  display-only pass-throughs of those helpers. Rendering the live trigger node needs a
  `<SvelteFlow>` provider for its `<Handle>` (the existing `TriggerNode.test.ts` only exercises
  the no-Handle stale path for the same reason), so DOM coverage would add fragile scaffolding
  for no extra logic. Verified via typecheck + svelte-check + the autofixer; **no agent drove a
  browser** — a live `:5173` spot-check of the badge tooltip + both inspector hints + a real
  Monitor "Unrouted input" line is owed (consistent with prior slices).
- Group B's input-activity badges (S04) will sit beside these cross-reference hints after groups
  merge — S14 depends on none of B's code (both read `store`/`project` independently).
