# S07 — Settings update progress + in-app badge (Group C, #48)

## Summary
Update is now fully in-app. The Rust startup OTA check no longer opens a native dialog — it only
publishes `updateAvailable`/`updateVersion` onto `boot://status`. The web `UpdateControl` runs the
whole flow (check → available → install with a streamed progress bar → auto-restart); the settings
dialog embeds it and a top-bar `UpdateBadge` opens the same dialog. Install runs only on user action;
the web never auto-restarts (Rust restarts itself once the user-triggered download completes).

## Acceptance
- [x] Progress bar reflects streamed percentages — `UpdateControl.test.ts` drives `progressPct`
  0/40/100 through the bridge via a fake adapter; asserts `aria-valuenow` + `.fill` width.
- [x] Native dialog gone; startup yields badge only — `lib.rs` `check_for_update` dropped the
  `.dialog()`/`blocking_show` block (+ removed the unused `DialogExt`/`MessageDialogButtons` import),
  now calls `publish_update_available`; reducer folds it → `UpdateBadge.test.ts` badge-on-availability.
- [x] Flow reachable + identical from badge and settings — single `UpdateControl`; badge only calls
  `onOpen` → opens the settings dialog, so there is one install implementation.

## Gates
`pnpm typecheck` 0 errors · `pnpm test` 108 files / 1086 passed · `cargo check` (shared target) ok.

## Deviations
- The download-with-progress path was **moved** from `check_for_update` into `install_update_now`
  (which previously downloaded with no progress) — startup no longer downloads, so progress belongs
  on the user-triggered install. Behaviour preserved, relocated.
- No `pnpm ui-shot` capture: the S07 surfaces (progress bar, badge, install) are desktop-only and
  update-availability-gated — they can't render in the plain-browser dev server, and even the base
  `top-bar` shot renders empty headless here. Covered instead by the component tests above.
- Progress-bar markup kept local to `UpdateControl` (not added to the shared styleguide) to avoid a
  merge collision with S08 over `design-system.html`; it's token-styled and self-contained.

## Context pack for the lane orch (S07 ∥ S08)
- **Mounts:** `UpdateBadge` in `TopBar.svelte` right cluster (between `ShareInfo` and the Settings
  `IconButton`); `UpdateControl` inside `AppSettingsDialog`'s "Updates" `Field`. **Nothing at the
  app root** — no overlay/app-root touch (S08's lane).
- **Defensive `start()`:** yes — both `UpdateControl` and `UpdateBadge` call the idempotent
  `desktopBridge.start()` in their own `onMount`. When S08 adds the app-root `start()`, it stays
  idempotent (first caller wins); no conflict, safe to keep.
- **Shared file touched:** `boot-reducer.ts`/`.test.ts` (additive: folds `updateAvailable`/
  `updateVersion` from the boot event, sticky across ordinary events; empty-stage now treated as
  "no change"). `desktop-bridge.svelte.ts` lost the 3 legacy free functions (only S07 used them).
