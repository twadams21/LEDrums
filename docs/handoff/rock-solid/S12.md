# S12 — Kill echo re-fire + offline-gated sim (E, issue #50)

## Summary
One connected MIDI hit now fires **exactly once** — the authoritative fire on the server — instead
of the old 2–3× (client echo re-fire + local sim + server). The web sim is now a strict **offline
preview**: it resolves and renders only when the engine link is closed. When connected, every
outbound path forwards to the server and the client displays the server's frames / bus levels /
monitor diagnostics. This is the authority principle S13–S17 all build on.

- `onInput` echo no longer fires the sim (the loop is dead); MIDI-learn still works from any input
  source (hardware at the server, another client's echo).
- `forwardMidi` / `hit` / `fireSectionGraph` fire the local sim only offline; connected, they
  forward and stay silent locally.

## Context pack (READ THIS — S13, S14, S15, S17 build on it)

**Where the authority gate lives.** All in `apps/web/src/lib/trigger-lab/store.svelte.ts`:
- `forwardMidi` (WebMIDI → server): fires `fireRawMidiLocal` **only** when `this.link !== 'open'`;
  always sends `{t:'midi'}`. Learn (`applyMidiLearn`) runs regardless.
- `hit` (pad surface): `if (this.link === 'open') { client.send({t:'key',…}); return; }` else fire
  the sim. (`onInput` has no `key` echo branch, so a connected pad hit is a single server fire.)
- `fireSectionGraph` (keyboard): `if (this.link === 'open') { forward the graph's trigger source
  (key/midi/cc/osc); return; }` else fire the sim. **← S13 replaces this synthetic-source forward.**
- `receiveInputEcho(kind, value, note, channel)` (**new** private method; the `onInput` handler now
  delegates to it): applies MIDI-learn from any source, **never** fires the sim.

**The exact "connected vs offline" rule.** The **firing** gate is `store.link === 'open'`
(`link: 'offline' | 'connecting' | 'open'`). Fire the sim only when `link !== 'open'`. Do **not**
gate firing on `useServer` — that is a stricter *render* gate
(`useServer = link==='open' && serverModel && serverFrame`) used to choose the visualiser source
(`previewFrame`/`model`). The instant the link is open the server is the authority, even before its
first frame lands; use `link` for firing, `useServer` for rendering.

**What echo still does (do not "fix" these — they're load-bearing):**
- MIDI-learn from any input source, via `receiveInputEcho → applyMidiLearn`.
- Monitor display survives: the server broadcasts its authoritative resolution as `{t:'monitor'}`
  diagnostics (`input-resolved`, `graph-fired` — `voice-engine-host.ts` `monitorVoiceDiagnostic`)
  → client `onMonitor → addMonitor`. The `{t:'input'}` broadcast still fires server-side
  (`apps/server/src/handlers/voice-input.ts`) for the Monitor "input" line, and outbound sends
  still show via `onSend` (fires unconditionally in `WSClient.send`).
- Engine resolution is **UNCHANGED**: both-fire is kept by design (doc 03 §4;
  `engine.ts` `resolveGraphsForEvent`, `engine.test.ts:946-958` still valid). Do not make it
  exclusive — S14 adds the drum-link icon to make both-fire *visible*, not to remove it.

**What NOT to redo (belongs to later slices):**
- **S13** — the keyboard `fireGraph` intent message. `fireSectionGraph` connected still forwards a
  *synthetic* source (`{t:'midi'|'cc'|'osc'|'key'}`) the server re-resolves. S13 replaces that with
  `{t:'fireGraph', graphKey, velocity}` so the server fires the exact graph (no re-resolution, no
  both-fire ambiguity from the keyboard). The gate + forward site is the `if (this.link === 'open')`
  block in `fireSectionGraph`.
- **S14** — the `input-unrouted` engine diagnostic and the drum-link icon/tooltip. Not started here.
- **S15 — SECTION-RECALL DOUBLE-SPAWN IS NOT YET GATED.** S12 scoped only the *input/fire* paths.
  `setActiveSection` (store.svelte.ts ~L1261) still calls `this.sim.recallSection(look)`
  **unconditionally** (then sends `{t:'recallSection'}` when connected). Today that's harmless
  because the core engine doesn't spawn looks. **When S15 adds engine look-spawn, gate this sim
  recall on `this.link !== 'open'`** (reuse the exact `link === 'open'` convention above) or the
  sim + engine will double-spawn looks when connected. This is why S15 is "blocked by S12".
- **S17** — LayersDock server truth. **Known interim regression:** LayersDock reads local sim
  voices (`store.voices`), which are now empty when connected (the sim no longer fires). Bus-level
  meters still work (`busLevels` streams from `onStats`). S17 makes the dock read server voice
  state when connected. This is the intended trade (was: *wrong* voices shown that diverged from the
  visualiser; now: *no* per-voice list until S17; then: *correct* server voices).

## Deviations
- **Section-recall sim spawn left ungated (deliberate, in-scope-minimal).** S12's stated scope is
  the input/fire paths (MIDI forward, pad hit, section-graph fire) + the echo. `setActiveSection`'s
  `sim.recallSection` is a section-change path, not an input/fire path, and its gating interacts
  with S15's engine look-spawn — so it's handed off to S15 (see Context pack → S15). Gating it now
  would be pre-empting S15's design.
- **`hit` connected send still gated on local resolution being non-empty.** `hit` keeps its original
  `if (toFire.length === 0) return;` guard *before* the link branch, so a pad the local active
  section doesn't map sends nothing to the server (unchanged from pre-S12). Preserving existing
  behavior; not expanding scope.
- No UI surface changes (as expected for this plumbing slice).
