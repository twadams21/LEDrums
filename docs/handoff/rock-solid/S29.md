# S29 — Modifier graph layer (group H, issue #54)

## What changed
- **Graph model**: `NodeKind` gains `'modifier'` (inert in trigger-flow eval — `evalNode`
  returns `[]`); `GraphNode` gains `modifierId?`/`bypass?`; `GraphEdge` gains `toPort?: 'in'|'mod'`.
- **Chain resolution**: new pure `voice.resolveModifierChain(graph, node)` (walk `mod` edges →
  y-order → mod→mod front-to-back, cycle-guarded) — shared by the core engine (`eval-graph.ts`
  play case → `PlayAction.modifiers`) and the web sim (`sim.ts`), so offline preview == real output.
- **Wiring**: `canConnect`/`canReconnect` + store `connect`/`reconnect` carry `toPort`; mod wires
  are scoped (only from a modifier node, only into a play/modifier `mod` input) and validation
  never throws (5k-case fuzz). Drop-anywhere routes by SOURCE kind (modifier → `mod`).
- **UI**: palette adds modifier nodes; play + modifier nodes render a distinct `mod` input handle;
  mod wires are dashed `--role-mod`; play nodes show a resolved-chain count chip; new
  `ModifierNodeInspector` (type / bypass / params / per-param envelopes, reusing effect-param
  controls). Design system regenerated (modifier node + mod handle/wire + NodeCard face).
- **Persistence**: modifier node + `toPort` round-trip verbatim (JSON is trusted; no coercion
  needed — `makeNode`/`modifierNodeInit` seed the fields, hydrate is idempotent over them).

## Deviations
- Modifier node `env` (per-param envelopes) is AUTHORED + persisted now, but not yet APPLIED at
  render — `ResolvedModifier` carries no `env` (doc 10 wires modulation of modifier params). The
  inspector envelope buttons are live authoring; dormant until doc 10, as designed.

## Context pack for S30–S32
- **A new modifier = core only + free UI**: add `modifiers/impl/<id>.ts` (`ModifierDef {id, name,
  category, paramSpec, createState?, apply}`, S28 shape) + register in `modifiers/registry.ts`
  `ALL[]`. The palette, `ModifierNodeInspector` (param controls from `paramSpec.type`:
  number→Slider, enum→Select, bool→Toggle), node face (`modifierName`), chain resolution, and
  persistence all pick it up automatically — NO per-modifier UI step.
- Modifier `paramSpec` uses core `ParamSpec` (`type`, not the web `kind`); the inspector reads
  `spec.type` directly — no mapping needed.
- Categories drive nothing yet but the inspector sub-line; S32's "gallery/palette by category"
  is the only place that will want a category filter (add to `GraphPalette` items).
