# S15 — Engine section looks: spawn/release (E, issue #50)

## Summary
The core engine now **spawns and releases a section's per-bus "looks"** on `recallSection`,
structurally mirroring the offline sim (`sim.recallSection`). This closes the visualiser gap at
the root: a connected recall makes the server engine play the section's base/trigger/effect loop
effects, so the streamed frames finally match what the offline preview always showed. The store's
sim recall is now gated on the engine link (offline only), so the sim + engine never double-spawn
when connected — the last piece of S12's authority principle, extended to the section-recall path.

- **Engine parity** (core): `recallSection` → per bus, release prior non-oneshot voices, then spawn
  the named look as a looped kit-scoped voice. Deterministic, non-stacking, oneshots untouched.
- **Bridge** (web): `buildShow` carries each section's `looks` onto the `voice.Show` (deep-copied).
  Looks are the *existing* fixture-seeded data — S15 makes the engine consume them; **S16 authors them**.
- **Gate** (web): `setActiveSection` fires `sim.recallSection` only while `link !== 'open'`.

## Context pack (READ THIS — S16 launches directly on it)

### The spawn/release seam (where looks are realised)
All in `packages/core/src/voice/engine.ts`:
- **`VoiceBusEngine.spawnSectionLooks(sectionId)`** (engine.ts:369) — the whole behavior. Called from
  the `recallSection` branch of `processEvent` (engine.ts:214), right after the `section-recalled`
  diagnostic. For each bus in **`this.show.buses` order**: release the bus's prior non-oneshot voices
  (`releaseVoice`, loop/hold only), then — if `section.looks[bus.id]` names an effect — spawn it.
- **`lookAction(effectId, via)`** (engine.ts:394) — builds the `PlayAction`:
  `{ mode:'loop', scope:'kit', busId:'', params, env:{}, latchKey:null }`. `busId:''` means the pool
  spawns on the **effect's own home bus** (`a.busId || effect.busId`), *not* necessarily the looks-key
  bus. `params` = the `${effectId}:default` preset's params, else the effect's spec defaults
  (`lookDefaultParams`, engine.ts:404 — typed to voice `ParamValues`, since the effects-layer
  `defaultParams` returns the wider `ResolvedParams`).
- Reference implementation (kept identical on purpose): sim `recallSection` / `lookAction`
  (`apps/web/src/lib/trigger-lab/sim.ts:768-788`). The release-then-spawn order and the release/spawn
  bus asymmetry are mirrored verbatim so review is a diff. In practice a look effect's home bus equals
  its looks key (base effects on `base`, etc. — fixtures satisfy this), so the asymmetry never bites.

### Where looks live (the data model — important for S16)
There are **two** section arrays on `voice.Show`, sharing section ids but not shape:
- **`Show.sections: Section[]`** where `Section = { id, name, looks: Record<busId, effectId|null> }`
  (`packages/core/src/voice/types.ts:255-260`). **Looks live here.** `spawnSectionLooks` resolves the
  recalled section by `this.show.sections.find(s => s.id === sectionId)`.
- **`Show.songs[].sections: SongSection[]`** where `SongSection = { id, name, slots }`
  (types.ts:273-277) — drives per-pad **hit** resolution. Unrelated to looks.
- The `recallSection` event carries `{ songId, sectionId }`; the **same** `sectionId` keys *both*
  arrays. S16's authored model MUST keep a section's looks-id aligned with its slot-section id, or a
  recall activates the slots but finds no looks (or vice-versa).

### About "state prefixes" (the assignment mentions them — clarification)
Look voices are **not** graph-eval voices: they carry no `statePrefix`, and they do **not** touch
`seqIndex`/`lastPick`/`prng`. Determinism here comes entirely from (a) a fixed `PlayAction` given
`(section, effect, preset)` and (b) release-before-spawn on repeated recall. Don't go looking for a
per-section+bus state-prefix scheme — there isn't one, and looks don't need one. (The `statePrefix`
machinery is for the *hit* path's layered graphs.)

### How the show-builder bridges looks
`apps/web/src/lib/trigger-lab/show-builder.ts:54` — `buildShow` maps
`sections: source.sections.map(s => ({ ...s, looks: { ...s.looks } }))`. The `ShowSource.sections`
is the store's `sections` rune (`store.svelte.ts:204`, seeded from fixtures via
`seedLookSections()` → `structuredClone(SECTIONS)`, `store/seed.ts:68`). So **today looks are
fixture data** (`fixtures.ts:257-260`: intro/verse/chorus). The bridge deep-copies `looks` so the
sent `Show` is a true snapshot (the module's stated invariant). This is the single explicit map point
S16 extends when looks become authored.

### The connected recall path (end to end)
`store.setActiveSection(sectionId)` (store.svelte.ts:1261) → offline: `sim.recallSection(look)` +
`snapshot()`; connected: **skip the sim**, `client.send({t:'recallSection', songId, sectionId})` →
server `apps/server/src/handlers/voice-input.ts:32/74` → `voiceHost.applyInput({kind:'recallSection',
…})` → the engine's `spawnSectionLooks`. No server change was needed — the engine already received
`recallSection`; it just does more now.

## What NOT to redo (belongs to S16 / later)
- **S16 — LOOKS AUTHORING.** The **web** authored model `SetlistSection { id, name, graphs }`
  (`apps/web/src/lib/app/setlist.ts`) has **no looks field**; looks presently exist only on the
  fixture-seeded `store.sections` (`Section[]`). S16 adds `looks: Record<busId, effectId|null>` to the
  authored model (default `{}`), persistence coercion + idempotent migration
  (`persistence.ts` `coerceAuthored`/`migrateSongs` pattern), the `setLook(sectionId, busId, effectId)`
  store mutation (+ autosave/live-resync on the existing `setShow` re-send), and the SectionInspector
  "Looks" group (one effect picker per bus, None option). **Apply `/make-interfaces-feel-better`.**
  The engine is ready: whatever ends up in `Show.sections[i].looks` gets spawned. S16's real work is
  authoring → getting looks into `store.sections` (or reconciling the fixture `store.sections` with
  authored per-section looks) so `buildShow` bridges them.
- **S17 — LayersDock server truth (interim regression, now extended to looks).** Connected, the sim no
  longer recalls, so `store.voices` carries **no look voices** either — the dock shows nothing per-voice
  while the visualiser (server frames) shows the looks correctly. Bus-level meters still stream
  (`busLevels` via `onStats`). S17 makes the dock read server voice state when connected. Intended trade.
- **Don't** try to "fix" the release/spawn bus asymmetry in `spawnSectionLooks` — it mirrors the sim by
  design; changing it would diverge the two implementations this slice exists to align.
