# S20 — Colour batch 2: wash / base / utility / meter (Group F, issue #51)

## Summary

Colour batch 2 applies the S19 recipe to the wash/base/utility/meter effects, so they can make
white and pastels instead of only fully-saturated hues. Three effects gained a real `saturation`
param threaded into `hsvToRgb` (**radial-wash**, **wipe-3d**, **meter-eq**); five already exposed
and threaded saturation and just needed their contract + swatch confirmed (**solid-base**,
**breathing-kit**, **strobe**, **hue-rotate-kit**, **sidechain**). **temp-sweep** was the only
non-trivial one: it hardcoded amber/blue and had no colour param at all, so it now exposes its
warm/cool hue **endpoints** (a range) plus a shared saturation.

Swatch behaviour follows S19 automatically — any effect carrying `hue`+`saturation`+`brightness`
gets the play-node ColorSwatch with no per-effect UI. radial-wash, wipe-3d, meter-eq, solid-base,
breathing-kit, strobe and sidechain now get it. The two intrinsically multi-colour effects stay
swatch-less: **hue-rotate-kit** (base hue + vertical spread) and **temp-sweep** (warm/cool
endpoints). Defaults reproduce the pre-S20 output byte-for-byte (saturation default 1; temp-sweep
warm 30 / cool 210).

No `ParamValue` re-widening (S18 owns that), no `mapParamSpec` change, no ColorSwatch change — so
no `docs/design-system.html` regeneration. Core-only + tests + audit table.

## Context pack (brief — batches are parallel siblings)

- **Recipe unchanged from S19.** Add `saturation` after `hue` (default 1), read
  `const sat = pnum(params, 'saturation', 1)`, swap the hardcoded `hsvToRgb(hue, 1, …)` →
  `hsvToRgb(hue, sat, …)`. Swatch is automatic once hue+saturation+brightness all exist.
- **Multi rule (matters for S21/S22).** An effect that spreads hue across space/time and has no
  single meaningful base hue is *multi*: expose saturation + a range/offset scheme, no single
  `hue` key → no swatch. In S20 that was `hue-rotate-kit` (baseHue + `ky`) and `temp-sweep`
  (warm/cool endpoints). But an effect with a real base `hue` + a `hueSpread` (like `synced-hoops`
  in S19, and `meter-eq` here) is NOT multi — it keeps a single `hue` and gets the swatch.
- **Already-compliant effects need no code.** solid-base, breathing-kit, strobe, hue-rotate-kit,
  sidechain already exposed + threaded saturation; verify before editing so you don't churn them.
- **Shared files with siblings:** the audit table (`effect-params-audit.md`) and
  `effects.test.ts` are extended by S21 (batch 3) too — S20 added its own `describe('S20 …')` block
  and appended rows only under the Batch 2 heading, leaving S19's/S21's regions untouched.

## Deviations

- **temp-sweep given warm/cool hue endpoints, not just saturation.** It had zero colour params
  (hardcoded `WARM_HUE`/`COOL_HUE` consts), so adding only saturation would still leave the warm/cool
  colours unpickable — a weak "colour pass". Exposing the two endpoints as params (the range scheme
  the recipe prescribes for multi effects) is the smallest change that makes it genuinely
  controllable; defaults reproduce the old amber↔blue exactly. Marked *multi* / swatch-less.
- **meter-eq classified swatch-worthy, not multi.** It spreads hue across hoops via `hueSpread`, but
  it retains a meaningful base `hue` — the exact `synced-hoops` shape S19 gave a swatch — so it keeps
  a single `hue` + gets the swatch rather than a range scheme. Consistent with the batch-1 precedent.
- No touch-list files outside doc 05's list; no `ParamValue` widening (S18), no `mapParamSpec` /
  ColorSwatch / design-system change.
