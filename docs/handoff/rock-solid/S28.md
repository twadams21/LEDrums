# S28 ‚Äî Modifier engine core + Trail (group H, issue #54)

## What changed
- New `packages/core/src/modifiers/` (mirrors effects/registry): `ModifierDef`/`ResolvedModifier`
  types, registry, and `applyModifierChain` ‚Äî the compositor's only seam into modifiers.
- First modifier: **Trail/Decay** (temporal smear; per-voice accumulator; `decayMs` + `mode` add/max).
- Voices carry a resolved chain (`voice.modifiers`) + per-voice state (`voice.modState`); spawn copies
  `PlayAction.modifiers` and resets state per voice (same lifecycle as `genState`).
- Compositor applies the chain between render and blend on **both** paths ‚Äî generator bridge (on
  `genScratch`) and pattern voices (scratch entered only when modified; zero-alloc hot path preserved).
- Web sim (`render.ts`) mirrors via the same core runner (generator + pattern paths).

## Deviations
- Chain-order is proven with two **Trail instances** (distinct decay+mode), since S28 ships only Trail;
  order-dependence is real (each link has its own accumulator). A 2nd modifier type is S30's.
- Sim modifier math runs in 0..1 float (core's space); the sim's pattern path is 0..255-rounded, so
  byte-parity was never exact there pre-S28 ‚Äî the **generator path is the byte-parity surface** and
  carries the parity test. Sim parity is asserted behaviourally (the S25/S26 pattern).
- `ResolvedModifier` carries no `env` yet ‚Äî per-param modulation of modifier params is doc 10 territory.
