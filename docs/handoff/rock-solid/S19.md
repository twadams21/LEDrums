# S19 — Colour batch 1: swatch + hit/trigger effects (Group F, issue #45)

## Summary

Effects can now make white and pastels, not just fully-saturated hues. The write-through
**colour swatch** lands in the play-node inspector: pick a colour and it decodes to
hue/saturation/brightness and drives the three sliders (and vice-versa — move a slider, the
swatch follows). Under the hood the eight hit/trigger effects that hardcoded `saturation = 1`
now expose a real `saturation` param, so saturation 0 desaturates a lit effect to white.
`colour-melody` already had it. This slice also establishes the colour-pass **recipe** and the
**audit table** the three mechanical batches after it (S20/S21/S22) follow without rediscovery.

## Context pack — THE COLOUR-PASS RECIPE (S20/S21/S22 follow this verbatim)

The batches after S19 are wide-but-shallow. Do NOT redesign — repeat these steps per effect.

**1. Add the param + thread it (core `impl/<effect>.ts`).**
   - Colour effects: add `{ key: 'saturation', label: 'Saturation', type: 'number', default: 1, min: 0, max: 1, step: 0.01 }` in `paramSpec` right after `hue`; add `+brightness` the same way if missing.
   - In `render`, read `const sat = pnum(params, 'saturation', 1);` and replace every `hsvToRgb(hue, 1, v)` with `hsvToRgb(hue, sat, v)`. That literal `1` in the S-slot is the whole bug — grep each file for `hsvToRgb(` and `, 1,`.
   - **Multi-colour effects** (rainbow-flow, hue-rotate-kit, and any effect that spreads hue across space/time): a single `saturation`/`hue` is wrong. Expose `saturation` + a **range/offset** pair (e.g. `hueRange`/`hueOffset`) instead of a single `hue`, and thread `sat` into every `hsvToRgb`. Mark these `*multi*` in the audit — they get NO generic swatch (see step 3).
   - Defaults MUST preserve current output: `saturation` default 1 reproduces the old `hsvToRgb(hue, 1, …)` exactly (verify with an existing golden / the `all effects` parity test).

**2. The swatch is automatic — no per-effect UI.**
   `PlayNodeInspector.svelte` renders `ColorSwatch` for ANY effect whose params include all of
   `hue` + `saturation` + `brightness` (see `COLOR_KEYS` / `hasColorSwatch` there). So step 1 is
   all that's needed to light up the swatch. Multi-colour effects deliberately lack a single
   `hue` param, so they skip it (correct — a picker can't represent a rainbow).
   - Swatch seam: `lib/ui/ColorSwatch.svelte` — props `hue`/`saturation`/`brightness`/`modulated`/
     `onChange(hsv)`; write-through is `applyColor(hsv)` in the inspector →
     `store.setParam(node, 'hue'|'saturation'|'brightness', …)`. `modulated` = any of the three
     `store.isEnveloped(node, key)` → the swatch shows the BASE colour + an env badge, never animates.
   - Colour maths live in core `color.ts`: `hsvToHex`/`hexToHsv` (compose `hsvToRgb`+`rgbToHex` /
     `hexToRgb`+`rgbToHsv`). Reuse; don't reinvent.

**3. Golden test (core `effects.test.ts`).**
   Extend the `describe('S19 colour batch 1 …')` block's `cases` array — one row per effect:
   `{ name, run: (m) => render(effect, m, ctx(m, { …enough to light pixels… }), { hue: 120, saturation: 0, …state }) }`.
   The shared assertion (`scanLit`) requires **lit > 0 AND every lit pixel achromatic** (r===g===b).
   Trigger effects need a `trig(...)`; base/texture effects need a `transport`/`timeMs`; stateful
   effects pass `effect.createState!(m)`. Copy the S19 rows as templates. (S20+ may rename the block.)

**4. Audit table.** Append rows to `docs/handoff/rock-solid/effect-params-audit.md` under your
   batch's PENDING heading (flip it to ✅ DONE): `| Name | id | Cat | before | after | Swatch |`.
   S22 must leave all 41 registry effects (`packages/core/src/effects/registry.ts`) accounted for
   — the tally at the bottom is the checklist (batch sizes 9/9/13/10).

- `ParamValue` was NOT widened — the swatch is UI-only, persistence/protocol carry only numbers
