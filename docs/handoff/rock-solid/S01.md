# S01 — Graph editor hardening (A, issue #46)

## Summary
The trigger-graph editor can no longer silently blank the canvas until refresh. Every
xyflow event callback (connect / connect-end / reconnect / delete / drag) now runs inside an
error boundary that reports the fault as a Monitor `error` event **and self-heals** by resetting
the projection cache and rebuilding from the store — no page refresh needed. A node whose live
model can't be resolved renders a **visible dashed "Stale node" placeholder** (with its id) instead
of a blank card. The projection cache is exception-safe (reset on graph-switch/error, write-through
only on success) with a dev-mode desync assertion, and wiring validation is proven never to throw.
All four incident-09 candidate mechanisms are either eliminated wholesale or made observable.

## Context pack
**H (S29) and I build on this hardening.** What was touched, the new invariants, and what NOT to redo:

- **New seam — projection cache lifecycle** (`apps/web/src/lib/app/views/trigger-flow-projection.ts`):
  - `resetProjectionCache()` — deliberate reset to the empty sentinel, distinct in *intent* from
    `emptyTriggerProjectionCache()` (identical structure). Use it on any error path or graph-open.
  - `projectionDesyncIds(flowIds, graphIds)` — pure diagnostic returning rendered ids missing from
    the store graph. This is the desync telemetry (candidate 1 vs 3). Reuse it if you add graph views.
  - **Invariant:** the cache is written through **only on a successful projection**. The pure
    `projectTriggerFlowNodes` returns `{ nodes, cache }`; the caller must not assign the cache before
    a successful return. If you add fields to the cache, keep this discipline.
- **New seam — flow callback error boundary** (`apps/web/src/lib/app/views/flow-guard.ts`):
  - `guardFlowCallback(where, fn, onFault)` — pure, framework-free wrapper. `PatchGraphView` has the
    same drop-anywhere wiring and is a **candidate for the same treatment** (see Deviations — I did
    not touch it, in scope). If you harden it, reuse `guardFlowCallback` + `store.reportError`.
- **Store seam** (`store.svelte.ts` → `reportError(source, label, detail?)`): the ONLY public way to
  emit a Monitor `error` event (`addMonitor` stays private). `source` groups faults (`trigger-graph`).
- **View wiring** (`TriggerGraphView.svelte`):
  - `guard(where, fn)` binds the boundary to the view's `reportGraphFault` (console + `store.reportError`)
    + `forceRebuild` (cache reset → rebuild nodes+edges). Every callback passed to `GraphCanvas` is
    wrapped. **Do not** pass a raw callback to `GraphCanvas` — wrap with `guard`.
  - `rebuildNodes()` resets the cache when `graphKey` changes **before** projecting, catches a
    projection throw (reset cache + report + keep last-good render — never blank), and runs the
    dev-only desync assertion (`import.meta.env.DEV`).
  - `openGraph()` resets the cache on graph-open.
- **Blank-proof node** (`TriggerNode.svelte` + `NodeCard.svelte`):
  - `NodeCard` gained a `stale` prop (dashed `--warn` card). It's now a documented styleguide state
    (`SectionComposites` → "Node card — states"); regenerate `docs/design-system.html` via
    `pnpm design-system` if you touch it.
  - `TriggerNode` renders the placeholder (warning icon, `title="Stale node"`, `sub=id`, **no handles**)
    when `store.selectedGraph?.nodes.find(id)` is null. `kind` still comes from the flow-node `data`
    payload (always present), so the icon/tint survive a model loss.
- **Wiring validation** (`store/graph-wiring.ts`): unchanged — the fuzz suite **proves** `canConnect` /
  `canReconnect` already never throw (unknown/self/cross-graph ids, port mismatches). No source change
  was needed; do not "defensively" wrap these — they are pure predicates by design.
- **What NOT to redo:** the component-field (not module-level) projection cache is intentional (a
  module-level cache was the incident's root-cause shape — do not reintroduce one); the `untrack`
  around projection + position reads is load-bearing (prevents drag-triggered rebuilds).

## Acceptance
- [x] **Injected throw in a flow handler → Monitor error + clean next projection.**
  `guardFlowCallback` catches the throw and routes it to `onFault` (proven in `flow-guard.test.ts`);
  the view's `onFault` = `reportGraphFault` (→ `store.reportError`, proven to emit a Monitor `error`
  event in `store.report-error.test.ts`) + `forceRebuild` (cache reset → rebuild).
- [x] **Projection error paths leave the cache reset; same-id nodes across graph switches rebuild
  from the new graph.** `rebuildNodes` resets the cache on new graphKey and on catch; the pure
  regression test (`trigger-flow-projection.test.ts`) shows a same-id node switching graphs rebuilds
  from the new graph (kind + position + cache re-key) and that `resetProjectionCache` breaks reuse
  (extends PR #37).
- [x] **A NodeCard whose id is missing from the store renders the placeholder (jsdom).**
  `TriggerNode.test.ts` — three scenarios (empty graph, other-nodes-but-not-this-id, no selected
  graph) all render the visible "Stale node" card (`.card.stale`, id surfaced), never a blank card.
- [x] **Wiring validation never throws for unknown ids / self-connects / cross-graph ids / port
  mismatches (pure fuzz).** `graph-wiring.test.ts` — 2×5000-iteration deterministic fuzz asserting
  no throw + boolean verdict, plus targeted dup/cycle/direction cases.
- [x] **Full sweep green (typecheck + tests).** See Gates.

## Deviations
- **`store.svelte.ts` (outside the incident touch-list):** added a public `reportError()` seam so the
  view can reach the Monitor bus (`addMonitor` is private). Minimal, additive.
- **Extracted `flow-guard.ts` (new file):** the error boundary was pulled out of the view into a pure
  helper so acceptance #1's throw→report→heal contract is unit-testable without driving xyflow. The
  view's inline `guard` now delegates to it.
- **`.gitignore`:** added `dist-design-system/` — the required `pnpm design-system` regenerate step
  leaves that build-artifact dir untracked (only `dist/` was ignored). One-line hygiene fix.
- **`PatchGraphView` NOT hardened:** it shares the drop-anywhere wiring and is a natural next target
  for `guardFlowCallback`, but it is outside the S01 surface (trigger graph). Flagged for H/I or a
  follow-up. No functional change made there.
- **Root cause still not *convicted*:** per doc 09 this slice is instrument-first. The hardening
  eliminates candidates 1 & 2 wholesale and makes 3 observable (dev desync assertion + Monitor
  errors); no single candidate was reproduced to convict, which is the expected outcome.
