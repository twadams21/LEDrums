# S13 — Keyboard fireGraph intent message (E, issue #50)

## Summary
The computer-keyboard performance path (keys 1–9 → the active section's graphs) now sends a new
`{t:'fireGraph', graphKey, velocity}` **intent** message when connected, instead of a *synthetic*
MIDI/OSC/key source the server re-resolved. The server plays **precisely that graph, once** — no
zone-map, no direct-binding re-resolution, so the old keyboard **triple-fire** (local sim + server
re-resolve + echo re-fire) is gone. A MIDI-bound section graph is the exact case that used to fire
3×; it now fires 1×. Offline, the keyboard still fires the local sim unchanged.

This builds directly on S12's authority principle: connected, the server is the sole resolver and
the client stays silent locally. S12 left `fireSectionGraph`'s connected branch forwarding a
synthetic source (the documented S13 hand-off); this slice replaces exactly that block.

## Context pack (brief) — for S14/S15/S17 and reviewers

**The wire contract (SSOT `packages/protocol`).** New client→server message
`{ t:'fireGraph'; graphKey: string; velocity: number }` (velocity 0..1). Whitelisted in the
server's `CLIENT_TYPES` (`ws-protocol.ts`) — a message type absent from that set is rejected by
`decodeClient`, so any *new* client message must be added there too.

**Where the fire-by-key lives (core, pure + deterministic).** `engine.ts` gained a `fireGraph`
`InputEvent` kind (carrying `graphKey`) handled by `processFireGraph`: it looks up
`show.graphs[graphKey]` and fires **exactly** that graph with `path:'fire-graph'`, emitting the
same `input-resolved` + `graph-fired` diagnostics as any fire. A stale/absent key emits
`graph-missed` with the new reason `'no-such-graph'`. It deliberately does **not** call
`resolveGraphsForEvent` — no zone-map, no direct-binding, no both-fire. `sourceDrumId` is derived
from the graph's own authored `drum` trigger source (so drum-scoped effects target the right drum);
a midi/osc-sourced graph carries `sourceDrumId:''`, identical to the server's existing direct path.

**The seam chain.** web `fireSectionGraph` (connected) → `{t:'fireGraph'}` → server
`handleVoiceInput` → `voiceHost.applyInput({kind:'fireGraph',…})` → `toInputEvent` → engine
`processFireGraph`. The engine's diagnostics reach the Monitor via `voice-engine-host`'s
`monitorVoiceDiagnostic` (unchanged) → `{t:'monitor'}` broadcast.

**Monitor surfaces for a keyboard fire (all broadcast to every client):** (1) `monitorInput`
(main.ts) emits a server-side `Fire graph <key>` ingress line; (2) the engine emits
`Graph resolved`/`Graph fired <key>` diagnostics. The handler does **not** broadcast a `{t:'input'}`
echo (unlike `key`/`midi`) — a fireGraph has no note/address to echo for MIDI-learn, and the two
lines above already surface it. See Deviations.

**Untouched by design (per assignment scope):** engine `resolveGraphsForEvent` both-fire is
**unchanged** (S14 makes it *visible* via the drum-link icon; do not make it exclusive). No
inspector/node-face work (S14). No engine look-spawn / section-recall gating (S15) — the S12
`setActiveSection` sim-recall note still stands for S15.

## Deviations
- **No `{t:'input'}` echo broadcast from the `fireGraph` handler** (the `key`/`midi`/`osc` handlers
  each broadcast one). A `fireGraph` carries no note/address, so there is nothing to echo for
  MIDI-learn, and the fire is already surfaced to every client by the server-side `Fire graph <key>`
  ingress line (`monitorInput`) + the engine's `Graph fired` diagnostic. Broadcasting an
  `{t:'input', kind:'midi'}` with no note would be a redundant, mislabelled monitor line. The
  handler test asserts on the `Graph fired` diagnostic (the real "one fire") rather than an input
  broadcast — a stronger assertion for the "exactly one fire" acceptance than counting echoes.
- **`sourceDrumId` for drum-scoped effects is derived from the graph's own `drum` trigger source**
  (in core `processFireGraph`), matching the offline sim's `sourceDrumIdForTriggerSource` for the
  common section-graph case. midi/osc-sourced graphs fire with `sourceDrumId:''` — the same as the
  server's pre-existing direct-binding path, so nothing regresses.
- **Removed the now-dead `midiVelocity()` store helper** — its only caller was the synthetic-MIDI
  branch of `fireSectionGraph` that S13 deletes.
- **Engine both-fire left unchanged** (assignment scope; S14 owns visibility, not exclusivity).
