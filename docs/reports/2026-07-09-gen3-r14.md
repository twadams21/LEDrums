# R14 — Fan-in to one Effect coalesces to a single firing (#93)

Branch `gen3r/r14-fanin-coalescing`. Tests-first, per the spec mandate.
Defined against R13's settled temporal model (`docs/reports/2026-07-09-gen3-r13.md`).

## What changed

Multiple flow edges converging on **one Effect node** now emit a **single voice
per trigger**, not one voice per incoming edge — fan-in no longer doubles (or
N×'s) the brightness. A delayed branch re-arriving at the same Effect drains in a
**separate** eval batch and remains a **distinct temporal firing** under the R13
timeline model — it is not folded into the immediate one.

### Root cause

In `evalGraphGen3FromPlan` the `effect`/`play` case looped over its incoming
bucket entries and pushed the play draft **once per entry**:

```ts
for (const entry of sourceEntries) pushKids(node, { kind: 'play', play: draft }, entry.latchKey);
```

But the play draft is a **pure function of the node** (`makePlayDraft` reads only
the node's effect/params/scope/mods) — the entries carry nothing but the latch
key. So two converging edges produced two byte-identical drafts → two output play
actions → two voices stacked on the same pixels → double brightness. This is
exactly the overlap double-count R13 explicitly handed to R14.

### Fix (core seam — `packages/core/src/voice/eval-graph.ts`)

A per-eval-call `firedEffects: Set<string>` guards each Effect/play node to a
single firing. The first time the node is processed it emits one draft (choosing
the first latched entry's key, mirroring the Mix collector's
`entries.find(e => e.latchKey)`); subsequent waves for the same node id are
skipped. Because a delayed drain re-enters via a **fresh** `evalChildren` call
(new `firedEffects`), delayed re-arrivals stay separate temporal firings. `delay
0` folds inline into the same batch, so it too coalesces to one firing.

Coalescing is deliberately scoped to the Effect **node**, not the Output
collector: two *distinct* effects wired into Output are genuine multiplicity and
must both fire — verified by test.

### Offline preview (`apps/web`)

No sim code change: the sim delegates Gen3 eval to core (R16), so it inherits
coalescing directly. A parity test asserts the preview follows it end-to-end.

## Tests

- **`packages/core/src/voice/eval-graph.fanin-coalescing.test.ts` (5):** two edges
  into one Effect → one firing; distinct effects into Output are *not* coalesced;
  velocity-invariant; delayed re-arrival fires separately on drain; `delay 0`
  inline still coalesces.
- **`apps/web/src/lib/trigger-lab/sim.fanin-coalescing.test.ts` (2):** two paths
  into one Effect → one voice; a delayed re-arrival spawns its own voice (born at
  the drain — lifecycle-robust vs decay/replace).
- **Regression floor stays green:** R13 delay-timeline (core 6 / sim 3) and the
  full Mix suite (17) unchanged.

## Deviations & follow-ups

- **None.** Scope held to `packages/core/src/voice/eval-graph.ts` + two new test
  files. No UI change (no ui-shot needed, per the ticket). Stayed clear of the
  inspector components (R26) and store wiring (R04) per the sibling note.
- Coalescing is keyed by node id within an eval call, so it is robust to an
  Effect fed downstream of a deferred collector (Mix→Effect) — that topology also
  fires the Effect once — though the common case is the direct two-edge fan-in.
