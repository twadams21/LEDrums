# R01 — Vanishing/reappearing wire bug (GH #80)

**Branch:** `gen3r/r01-wire-bug` · **Status:** done (root cause fixed + regression test)

## Root cause (stated for the closing commit)

**A wire drawn into a Mix node silently vanished until a page refresh because the
Mix node's projection signature ignored its incoming flow-edge set.**

The Trigger Graph canvas projects the store graph into xyflow. To preserve xyflow's
measured handle bounds (and avoid the "adding a node drops wires / snaps position"
bug), `projectTriggerFlowNodes` **reuses** the previous flow-node object for any node
whose `triggerNodeSignature` is unchanged.

A **Mix node renders one input handle per incoming flow edge** —
`mix-edge:<edgeId>` (`graph-to-flow.ts` → `mixRowHandleId`, rendered in
`TriggerNode.svelte`). But `triggerNodeSignature`'s `mix` case encoded only
`mixBlendMode`. So when the user dropped a wire onto a Mix node:

1. `store.connect` pushed the edge → `edgeSig` changed → `rebuildEdges` ran and the
   new edge referenced target handle `mix-edge:<newId>`.
2. The Mix node's signature was **unchanged** (no node field changed), so
   `nodeSig` didn't change and the projection **reused the stale Mix flow-node
   object** — whose `measured.handleBounds` had no entry for the new row handle.
3. xyflow had nowhere to attach the wire → it **vanished**.
4. On refresh the Mix node was created fresh, all row handles measured from the
   start → the wire **reappeared** (it had persisted correctly all along).

This is the "duplicate/mis-projected edge the projection didn't render" candidate
from the brief — specifically, the Mix node is the only node kind with a *dynamic,
per-incoming-edge* handle set, so it was the only target where a freshly-drawn wire
had no measured handle. Wires into fixed-handle nodes (effect/output/switch bands)
always attached fine. The `linked`-mod path (`modInputs`) already folded its dynamic
handles into the signature (see the existing `modInputs` in `base`) — Mix was the
missing symmetric case.

The other candidate — *"a failed add masked by refresh-time auto-wiring"* — was
**ruled out** as the vanish mechanism: the legacy leaf→Output auto-wire in
`normalizeTriggerGraphToGen3` runs only for `version !== 3` graphs, and the store
normalizes every graph to v3 at construction/show-load, so it cannot re-add a wire
in the steady-state authoring path. It remains a real (but *silent*) refresh-time
normalisation — surfaced for R02 below.

## Fix

`apps/web/src/lib/app/views/trigger-flow-projection.ts`
- `triggerNodeSignature(n, graph?)` now takes the graph; the `mix` case folds in
  `mixRowSetSignature(graph, n.id)` — the sorted **set** of incoming flow-edge ids
  (order-independent, so a source-node drag that only reorders rows doesn't churn;
  modulation/`mod` wires excluded). Adding/removing a wire into a Mix node now
  changes its signature → the node rebuilds → xyflow re-measures the row handles →
  the new wire attaches immediately, no refresh.
- `projectTriggerFlowNodes` passes `graph` into both signature calls.

`apps/web/src/lib/app/views/TriggerGraphView.svelte`
- `nodeSig` passes `store.selectedGraph` so a wire-add into a Mix node also trips
  the node-rebuild `$effect` (previously only `edgeSig` fired).

## Refresh-time normalisation seam (for R02 / GH #82)

`apps/web/src/lib/trigger-lab/store/hydrate.ts` — the hydrate boundary discards
`normalizeTriggerGraphToGen3(...).issues`, so Gen3 migration, Output-anchor
insertion, legacy leaf→Output auto-wire, and duplicate/dangling-edge drops happen
**silently**. Added a clean SEAM comment on `migrateGen3Graph` documenting that
R02 should thread `.issues` up to the store to raise one system-action toast per
event. **No toast UI built here** (per the brief — R02 owns it).
