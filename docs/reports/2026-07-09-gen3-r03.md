# R03 — Connection-time validation (GH #82)

**Branch:** `gen3r/r03-connection-validation` · **Phase 1.1**

While dragging a wire, the wire-in-progress now turns **red, dotted, and dull** the moment the
pointer crosses an invalid target; releasing over an invalid target shows a **toast naming the
reason** (direction / cycle / duplicate). No wire silently vanishes. Verdicts come from the
existing store connection validator (R01) — no parallel validator — and reuse the R02 toast host.

## What changed

- **Store seam — reason-returning validator** (`store/graph-wiring.ts`):
  `classifyConnection` / `classifyReconnect` return `WireRejection | null`
  (`direction` | `duplicate` | `cycle`), checked in that precedence. `canConnect` / `canReconnect`
  are now thin `=== null` wrappers over them (behaviour unchanged; still pure + total, never
  throws). `store.connect` / `store.reconnect` return the rejection reason (`null` on success or a
  viewer/no-graph no-op) so the view can surface *why*.
- **Reason copy** (`store/wire-toasts.ts`): `wireRejectionMessage(reason)` — one plain-language
  sentence per reason, exhaustive over the union.
- **View emission** (`views/TriggerGraphView.svelte`): `onConnect` / `onReconnect` / `dropConnect`
  push one `error`-tone toast on a refused wire.
- **In-drag styling** (`views/WireDragValidity.svelte` + `GraphCanvas.svelte`): a `useConnection`
  tracker (child of `<SvelteFlow>`) hit-tests the node under the pointer during a drag and asks the
  view's read-only `validateDrop` predicate whether the drop would be accepted — driving a
  `.wire-invalid` class that renders the connection line red / dotted / dull. Drop-anywhere-aware
  (xyflow's native `isValid` only fires over a handle, not a node body); honours the precise handle
  when the pointer is inside one. `validateDrop` mirrors `dropConnect`'s routing but never mutates
  (a `predictParamPort` variant replaces the auto-exposing `paramPortFor`).
- **ui-shot** (`views/wire-preview.svelte.ts` + `shot-seam.ts`): a `wire-invalid` state op pins a
  static stand-in of the drag-only wire (same xyflow classes → identical styling) so the state is
  screenshot-able. No `shots.json` entry (ad-hoc per conventions).

## Deviations / notes

- The in-drag `validateDrop` predicate is a read-only best-effort mirror of the live drop routing;
  the store's verdict **on release** (which drives the toast) is authoritative. Any divergence is
  confined to exotic modulation auto-expose cases and only affects the pre-release colour, never
  the committed wire or the toast.
- Added the reason toast on the **reconnect** (edge-end drag) path too, for parity — the brief
  centred on the connect path.
- Stayed out of `packages/core` eval (R13) and the inspector components (R25).
