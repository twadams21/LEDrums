# R02 — System-action toasts for migrations and auto-wires (GH #81)

**Branch:** `gen3r/r02-system-toasts` · **Status:** done

## What changed
Anything the app does on the user's behalf at hydrate now announces itself in **one
plain-language toast**, batched per user-visible load event. Reuses the existing toast
primitive (`lib/ui/toast.svelte.ts` / `ToastHost`) — no new UI component.

The emission is a **seam**, not toast calls sprinkled through store logic:

1. **Core (pure)** — `normalizeTriggerGraphToGen3` now also returns a
   `TriggerGraphSystemActions` summary (`migratedToGen3`, `autoWiredToOutput`) alongside
   `graph`/`issues`. This is the single place the Gen3 migration + Output auto-wire happen,
   so the facts are read straight off it (no before/after guessing).
2. **Web hydrate (pure)** — `sanitizeGraphsIntegrityWithActions` aggregates those per-graph
   facts across the whole map into one `SystemActionSummary`; `normalizeGraphs` returns it.
   Read off the **first** integrity pass, which is where migration actually lands.
3. **Emission layer** — `store/system-toasts.ts`: `describeSystemActions` (pure, testable copy)
   → `announceSystemActions` (pushes one `info` toast, injectable `push`).
4. **Store wiring** — `TriggerLab.normalizeGraphs()` calls `announceSystemActions(actions)` once.
   That method is the single choke point every load / adopt / show-switch funnels through, so a
   migration announces exactly once, batched. An already-Gen3 (no-op) hydrate yields an empty
   summary → stays silent (fresh boot is quiet; seed graphs are Gen3 from birth).

Kept strictly to the hydrate/migration + toast-emission seams so R01's parallel work in the
store connect/projection path merges cleanly.
