## API-preserving delegation
`store.svelte.ts` keeps its exact public surface as **thin accessor delegators + forwarders** to
`showsCtl`, so every existing `this.songs` / `this.activeShowId` / `this.resolvedSongs` / … call site
— including R24's section methods, the S44 paste/clipboard region, `toAuthored`/`applyAuthored`/undo,
and the engine `showSource` — keeps working **unchanged**. No consumer or test moved.
- get/set delegators: `activeShowId`, `songs`, `songRefs`, `activeSongId`, `songLibrary`.
- getter delegators (deriveds): `shows`, `activeShow`, `activeSong`, `resolvedView`, `resolvedSongs`,
  `songLibraryList`.
- forwarders: the 19 show/song/library-ref methods above.
- constructor calls `showsCtl.hydrateFromStorage()`; `onState`/`onShowLibrary`/`onSongLibrary` call
  `reconcileOnState`/`followShowLibrary`/`followSongLibrary`; the autosave tick calls
  `showsCtl.currentLibrary()` / `currentSongLibrary()` / `syncLibraryToServer()` /
  `syncSongLibraryToServer()`. Only the engine SHOW push (`syncShowToServer`, resolved-in refs) stays.

## Deviations
- One Svelte-5 idiom shift: `resolvedView` is `$derived.by(() => …)` (not `$derived(expr)`) so the
  reactive host reads live inside a closure — a field initializer that referenced `this.host`
  directly trips strict "used before init" (host is a constructor param). Lazy + memoized identically.
- `writeStoredLibrary` / `writeStoredSongLibrary` are exported from the controller module (the store's
  autosave debounce still writes through them); the `readStored*` boot helpers are module-private
  there. `readStoredKey` moved wholesale (the store's PIN/host-token readers use `sessionStorage`
  directly, so nothing else needed it).
- No behaviour change.
