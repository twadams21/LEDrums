## Verdict

**Changes needed — 1 blocking, 2 should-fix, 2 nits.**

The eval-seam semantics both tickets were tested against are correct and the
suite is green (core delay-timeline 6/6, fan-in 5/5 verified locally; reports
claim full `pnpm gates` green). Purity/determinism, `setShow` snapshot
clearing, delay-0 parity, fan-in coalescing, and sim↔core parity all hold.

**But the headline R13 scenario ships a newly-introduced render regression:**
during the delay-overlap window the still-live member is composited **twice**,
and no Phase 3 ticket owns the fix. R13's report disclosed this and attributed
it to R14; R14's scope and ACs never included it. This is invisible to the
test suite because every R13/R14 test asserts a single eval-call action list,
which structurally cannot observe two voices coexisting across ticks.

## Blocking

### B1 — Delay→Mix overlap double-counts the still-live member (double brightness), unowned across Phase 3
`packages/core/src/voice/eval-graph.ts:422-427` (R13 fold) + `voice-pool.ts:103-192` (spawn; no origin-based replacement)

**Mechanism.** When a delayed branch drains into a Mix, the R13 fold
(`eval-graph.ts:422`) re-includes still-live earlier members and the drain
spawns a **fresh** re-composed Mix voice. Nothing releases the *immediate* Mix
voice that already renders those same members — the only release path in
`spawn` is the mono-bus steal (`voice-pool.ts:109-111`), and layered/mix
content runs on **poly** buses (default; see the show fixtures throughout
`compositor.test.ts`, `engine.test.ts`).

Critically, the fold is gated on `isLayerLive(pad, 'a')`, and origin `'a'` is
live **iff** the immediate Mix voice `Mix[A]` is still active (A is never a
solo voice — it only exists wrapped in the composite). So the fold firing and
the old voice being alive are the *same condition*: whenever A is folded into
the drained `Mix[A,B]`, the old `Mix[A]` is guaranteed still alive → A is
guaranteed composited twice for the whole overlap window (including A's release
tail, since released voices stay `active` until reaped).

**Failure scenario.** Graph: immediate `A → Mix`, delayed `delay(100ms) → B →
Mix`, both effects on a poly bus. Fire once.
- t=0: spawn V1 = `Mix[A]` (renders A).
- t=100ms drain: `isLayerLive('a')` true (V1 alive) → spawn V2 = `Mix[A,B]`
  (renders A **and** B).
- Overlap window: V1 and V2 both alive → **A's pixels are ~2× brightness**,
  B's are 1×. This is *worse* than pre-R13 behaviour, where the drain produced
  `Mix[B]` and A was never doubled.

**Why blocking for the phase gate.** Phase 3's entire purpose is "engine
temporal semantics," and US13 ("a delayed branch behaves like a clip shifted on
a timeline") is the headline. The delivered path introduces a brightness
doubling in exactly that primary scenario. The eval-seam ACs of #92 technically
pass, so this is defensible as a fast-follow rather than a re-open — but it must
be an explicit, owned Phase 3 followup before this is called done, not left in
the R13→R14 handoff gap (see N1).

**Fix direction (for the followup ticket, not this review).** On a delayed
re-composition, release/replace the prior Mix voice for that (pad, mix-node)
rather than stacking a second composite — e.g. the drain steals the immediate
Mix voice by (pad, mixNodeId) origin before spawning, or the compositor treats
same-(pad,mix) composites as a single evolving voice. Needs an engine-level
test (see S1).

## Should-fix

### S1 — No engine/integration test covers the two-voice overlap; the regression is untestable at the seam tested
`packages/core/src/voice/eval-graph.delay-timeline.test.ts`, `.../trigger-lab/sim.delay-timeline.test.ts`

Every R13/R14 test asserts the action list of a **single** `evalGraph` /
`evalChildren` call. The double-count in B1 only manifests across two ticks
(immediate spawn, then a later drain spawn) at the voice-pool / compositor
level, which these tests never reach. Result: the suite is green while the
user-visible behaviour is wrong. Add an engine-level test that `setShow`s the
delay→Mix graph, fires, ticks past the delay, and asserts active-voice count /
composited brightness for the overlapping member. (This test is also the
acceptance gate for B1's fix.)

### S2 — `mixMemberSnapshots` liveness aliases across trigger instances on the same pad
`packages/core/src/voice/eval-graph.ts:422-426`, `voice-pool.ts:73-80`

`isLayerLive(pad, originNodeId)` scans by (pad, origin) only — it cannot tell
*which* trigger instance's voice is live. Two rapid fires on the same pad share
the pad key, so a delayed drain from trigger #2 will fold origin A back if
**trigger #1's** A voice is still alive, even after trigger #2's own A has
decayed. The snapshot map is also keyed per (pad, mix-node) and only reset on
`setShow` (`engine.ts:277`) — bounded, not a leak, but it retains stale member
drafts for the whole show lifetime, and a re-fire re-reads them (guarded by
`isLayerLive`, so behaviour is "aliased-live," not "stale-params"). Under the
timeline model this may even be acceptable, but it's non-obvious and undocumented;
worth a decision + comment, and it compounds B1 (aliased liveness → more folds →
more double-counts).

## Nits

### N1 — R13↔R14 handoff is inconsistent; the deferred double-count has no owner
`docs/reports/2026-07-09-gen3-r13.md:57-61`, `docs/reports/2026-07-09-gen3-r14.md:66-73`

R13's report states the immediate-vs-drained Mix double-count is "R14's job,
explicitly defined against your settled temporal model." R14's report lists
deviations "**None**" and its scope/ACs (#93) are strictly fan-in-to-one-Effect
— it never touched the Mix-voice overlap. Both reports are internally honest but
mutually contradictory: the deferred item fell through the crack. The spec's
Phase 3 has no 3.x line item for it either (3.1 timeline, 3.2 fan-in, 3.3
inspector). File B1 as the missing owner.

### N2 — Coalesced Effect drops per-edge latch keys silently
`packages/core/src/voice/eval-graph.ts:381-382`

Fan-in coalescing picks `newEntries.find(e => e.latchKey != null)?.latchKey`
and fires once. If two converging edges carry **different** latch keys (e.g. two
toggle paths into one Effect), only the first registers; the other latch key is
discarded. Almost certainly the right call (one voice ⇒ one latch), and mirrors
the Mix collector, but it's an unremarked behaviour change from the old
push-per-edge loop. A one-line comment noting the intentional drop would prevent
a future "why doesn't my second toggle release this?" bug hunt.

## What passed (verified)

- **delay-0 parity** — `delay 0` produces a byte-identical Mix composition to no
  delay and emits no pending fire (`eval-graph.delay-timeline.test.ts` 1–2;
  code path `eval-graph.ts:519-520` folds inline). ✔
- **Fan-in coalescing** — two edges into one Effect → one firing; distinct
  effects into Output stay multiple; velocity-invariant
  (`eval-graph.fanin-coalescing.test.ts` 1–3; guard `eval-graph.ts:379-380`). ✔
- **Delayed re-arrival = separate temporal firing** — drain re-enters via a fresh
  `evalChildren`/`firedEffects` (`engine.ts:681`, fan-in test 4–5). ✔
- **Core purity / determinism** — no wall clock, no unseeded RNG, no IO in the
  new paths; `firedEffects` is per-call, snapshots are engine-owned and passed in.
  ✔
- **Engine-owned state cleared on `setShow`** — `mixMemberSnapshots.clear()`
  (`engine.ts:277`); sim mirror clears on reset (`sim.ts`). No cross-show leak. ✔
- **Sim ↔ core parity** — sim delegates Gen3 eval to core and mirrors the snapshot
  map + `isLayerLive` + `pad`/`originNodeId` voice tagging; coalescing inherited.
  ✔
- **US15 inspector copy (3.3)** — `ContainerNodeInspector.svelte:113` states the
  canvas y-order stacking rule, consistent with the ascending-y, host-first sort
  (`eval-graph.ts:429-433`, `inputs[0]` = lowest y = topmost = base). ✔
