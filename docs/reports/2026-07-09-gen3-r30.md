## 1. Keyboard-shortcut interception seam

New module `apps/web/src/lib/app/shortcuts.ts` — ONE capture-phase seam that CLAIMS
browser-default combos from a **data** registry (`combo → action + description`):

- Matcher is pure + unit-tested: combo parsing (`'mod+shift+z'`), exact modifier
  matching (extra modifiers never collide), mac/win normalization via the existing
  `primary-shortcut.ts` (`mod` = Cmd on macOS, Ctrl elsewhere, mutually exclusive).
- `dispatchShortcut` guards against editable focus (input/textarea/select/contenteditable)
  unless an entry is `global`, then `preventDefault()` + `stopPropagation()` and runs the
  action. A `run()` returning `false` is a no-op that **falls through** (key not claimed).
- `describeShortcuts` projects the registry to render-ready data for a future help sheet.
- Module header documents interceptable combos (Ctrl/Cmd+D/S/K/O/P, F-keys) vs
  browser-reserved ones (Ctrl/Cmd+W/T/N, Ctrl+Tab).

Wired in `App.svelte`:
- **Undo relocated** — the inline `Ctrl/Cmd+Z → store.undo()` branch moved into the registry
  (no duplicate handler). *No redo exists* in the store (no redo stack / `redo()` method), so
  only undo was relocated — see Deviations.
- **Ctrl/Cmd+D = duplicate selected node** — `store.duplicateNode(node)` (a clean mutator
  already exists; no minimal impl needed), then selects the clone. A no-op unless a graph node
  is selected, so the Sections view keeps its own Cmd/Ctrl+D (duplicate **section**) when a
  section is selected — the two coexist because the seam only stops propagation when it handles.

**Files:** `lib/app/shortcuts.ts` (new), `lib/app/shortcuts.test.ts` (new, 12 tests), `App.svelte`.

- **Stray PIN-gated dev server** on :4321 (leftover tunnel session, not mine to kill) forced an
  isolated capture stack. No code impact.

  (not browser-default claims); left as-is to stay in scope.
