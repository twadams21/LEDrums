# R21 — Store split 2/5: extract the MIDI input/learn controller (#100)

**Branch:** `gen3r/r21-store-midi` · **HEAD:** `4e99f1e`

## What changed
Second slice of the trigger-lab store split. Extracted the **MIDI input + MIDI-learn**
machinery out of `store.svelte.ts` into a constructor-injected controller, following the
R20 `ControllerMonitor` prior art so the five slices read as one design.

New module `apps/web/src/lib/trigger-lab/midi-controller.svelte.ts` — `MidiController`:
- **Runes-owned reactive state:** `learnTarget`, `devices`, `available`, `unavailableReason`;
  private `handle` (the WebMIDI access handle).
- **Device layer:** `openInput(onEvent)` (opens WebMIDI, surfaces device list + availability,
  never throws — absent API / denied access → unavailable handle) and `release()` (store
  lifecycle stop: stop the handle, clear device state).
- **Learn arm:** `startLearn` / `cancelLearn` (viewer-gated arming) and `applyNoteLearn(note)` /
  `applyCcLearn(controller)` — the zone/trigger/cc-node bind logic, incl. the reserved-CC-0 skip.
- **Injected host** (`MidiControllerHost`): `isViewer()`, `getInputMap()`, `setInputMap()`,
  `setTriggerSource()`, `selectedGraphNodes()` — keeps the project/routing plumbing and
  graph-editing internals in the store.
- `MidiLearnTarget` moved here (its natural home) and **re-exported from the store** so the
  inspectors' import path is unchanged.

`store.svelte.ts` now **delegates** its public surface (API-preserving):
- `midiLearnTarget`, `midiDevices`, `midiAvailable`, `midiUnavailableReason` — getters.
- `startMidiLearn` / `cancelMidiLearn` — thin forwarders.
- `start()` opens MIDI via `this.midi.openInput((ev) => this.forwardMidi(ev))`; `stop()` calls
  `this.midi.release()`.
- `forwardMidi` / `receiveInputEcho` **stay in the store** and call `this.midi.applyNoteLearn` /
  `this.midi.applyCcLearn` once channel-gated.

## What deliberately stayed (scope boundary)
Per the brief ("routing fed by MIDI events vs the MIDI device/learn machinery — leave it in
place"): `forwardMidi`, `receiveInputEcho`, `fireRawMidiLocal`, `recordInputActivity` /
`inputActivity` (S04 badges, shared with OSC), the `midiChannel` derived + `setMidiChannel` +
`acceptsMidiChannel` channel filter, and the sim CC/note/OSC tables. These are entangled with
the offline sim, the shared activity map, and the local-fire preview — no clean MIDI-only seam.
Tests drive `internals(store).forwardMidi(...)` directly, so that method staying on the store is
also what keeps them **unmodified**.

> Architecture note: filed **#109** (wayfinder research) on whether MIDI should move to a
> server-side CoreMIDI service (Mac-only engine, advertised virtual port, external-source
> selection) vs today's browser WebMIDI-over-WS. That's a transport re-architecture, out of
> scope here; this extraction gives it one cohesive seam to rewrite behind.

## Seam for R22 (undo history)
The undo machinery is the next cohesive cluster, distinct from what moved here and untouched by
this slice: `undoLimit`/`undoStack`/`restoringUndo`/`suppressUndoSnapshot` (fields ~730–736),
`pushUndoSnapshot` (~1126, with the `restoringUndo`/`isViewer`/`suppressUndoSnapshot` guard),
`batchIntoCurrentUndo`, and `undo()`. Extract it the same way: an `UndoHistory` controller owning
the stack + flags (injected `toAuthored()`/`isViewer()`/a restore applier), the store delegating
`undo()`/`pushUndoSnapshot()`/`batchIntoCurrentUndo()` via forwarders. R22 starts from a clean base.
