- **`takeover` getter** — the active test pattern (or null in LIVE mode), read

  `currentTestPattern()`.

## Interpretation note (plan reconciliation)
The Phase-4 spec (spec line 56) names **monitor** and **controller test** as two
*separate* controllers. R20 pragmatically **bundled** the S49 test-pattern
(takeover getter + `setTestData` + `backToLive`) *into* `ControllerMonitor`,
because it shares the same WS-send + viewer gating. So by R22 the store no longer
held a cohesive "controller test" cluster — only thin delegators already pointing
at the monitor. R22's job was therefore to **carve the test-pattern concern out
of `ControllerMonitor`** into its own module, making the two named controllers
exist separately. The orchestrator confirmed this reading (A); the alternative —
extracting the local sim play/test-fire surface (`hit`/`fireSectionGraph`/
`fireRawMidiLocal`/`togglePlay`) — was explicitly ruled **out of scope** for R22
(that surface stays put; R23/R24 territory if anywhere).

## API-preserving delegation
`store.svelte.ts` keeps its exact public surface, re-pointed from `monitor` to the
new `controllerTest`:
- `controllerTakeover` — getter → `this.controllerTest.takeover`.
- `setControllerTestData(pattern)` / `backToLive()` — thin forwarders → `controllerTest`.
- New private `controllerTest` field wired with `currentTestPattern: () =>
  this.monitor.status?.testPattern ?? null`, so the takeover still tracks the
  monitor's server-authoritative status.
`ControllerMonitor` loses `takeover` / `setTestData` / `backToLive` (and its now-unused
`ControllerTestPattern` import) — it is now purely the monitor (status / candidates /
scanning + watch / discover / adopt / setAuth / identify).

Consumers unchanged: `OutputPill.svelte` and `PatchControllerInspector.svelte`
still read `store.controllerTakeover` / call `store.setControllerTestData`.
