## What changed
The FIFTH and FINAL slice of the trigger-lab store split — this closes spec 4.5. Extracted the
**section-arrangement** concern (S16 looks / U4 merged active section) into a constructor-injected
controller, mirroring R20 (`ControllerMonitor`), R21 (`MidiController`), R22 (`ControllerTest`), and
R23 (`ShowsController`) so every slice reads as a named controller.

New module `apps/web/src/lib/trigger-lab/sections-controller.svelte.ts` — `SectionsController` (+201).
It **owns** (Svelte 5 runes) the section-arrangement AUTHORING model for the active song:
- the ONE active/selected section pointer `activeSectionId` (U4 — the section you play IS the one you
  edit) + the transient `sectionClipboard` (in-app copy/paste scratch, not persisted);
- the `activeSection` / `sections` deriveds — read the active song through the host, so a
  referenced-library song's sections resolve just like a local one (S42);
- section CRUD, all funnelled through the private `updateActiveSong` chokepoint (the S2 viewer guard):
  `addGraphToSection` / `removeGraphFromSection` / `setSectionGraphs` / `moveSection` /
  `moveGraphPlacement` / `setLook` (S16) / `addSongSection` / `renameSection` / `removeSection` /
  `copySection` / `pasteSection` / `duplicateSection`, plus `insertSection` (the post-remap section
  insert the S44 system-clipboard paste performs — extracted from the store's `applyRemapResult`).

## Injected host (`SectionsControllerHost`)
The section edits mutate the `songs` rune that `ShowsController` owns (R23), so the seam reaches back
through a small host rather than owning songs: `isViewer` (S2 guard); `activeSong` (the deriveds +
`removeSection`/`copySection` reads — through R23's resolved `activeSong` getter); `activeSongId` +
`songs`/`setSongs` (the `updateActiveSong` immutable map, exactly as R23's report anticipated — R24
mutates `songs` through the store's delegators, not R23's internals); `linkOpen` +
`recallSectionLook` (the lone play-surface touch — `setLook`'s offline sim re-morph, which the store
performs via `sim.recallSection` + `snapshot`).

## API-preserving delegation
`store.svelte.ts` keeps its exact public surface as **thin accessor delegators + forwarders** to
`sectionsCtl`, so every `store.activeSectionId` / `store.sections` / `store.setLook(...)` / … call
site — components, the play surface (`hit`/`fireSectionGraph`/`setActiveSection`), the authored-state
swap (`toAuthored`/`applyAuthored`), and the S44 clipboard region — keeps working **unchanged**. No
consumer or test moved.
- get/set delegators: `activeSectionId`, `sectionClipboard`.
- getter delegators (deriveds): `activeSection`, `sections`.
- forwarders: the 12 section-CRUD methods above.
- `applyRemapResult`'s section branch now calls `sectionsCtl.insertSection(res.section)` (was a
  private `updateActiveSong` + `activeSectionId` set — behaviour identical).

## What stays in the store (the play surface, per the brief)
The split extracts the arrangement/authoring model, **not** the play surface. The store keeps:
`hit` / `resolveHitGraphsLocal` / `fireRawMidiLocal` (pad + raw-MIDI fire), `fireSectionGraph` (the
keyboard 1–9 section-graph fire), `setActiveSection` + `selectGraphInSection` (activate/select →
look recall + engine `recallSection`), and the sim/snapshot they drive. These read the section
state through the new delegators; `setLook`'s offline re-morph reaches the same sim via the host.

## Deviations
- `activeSection` / `sections` are `$derived.by(() => …)` (not `$derived(expr)`) so the reactive host
  reads live inside a closure — a field initializer that referenced `this.host` directly trips strict
  "used before init" (host is a constructor param). Same idiom R23 documented for `resolvedView`.
  Lazy + memoized identically.
- No behaviour change.

## Split complete — the store's remaining concerns (closes spec 4.5)
Five slices are now extracted into named controllers (Σ 1157 lines): `ControllerMonitor` (R20, 101),
`MidiController` (R21, 140), `ControllerTest` (R22, 54), `ShowsController` (R23, 661),
`SectionsController` (R24, 201). What remains in `store.svelte.ts` (3517 lines) is the **reactive
bridge + integration core** — the concerns that are genuinely central, not further-splittable clusters:
- **the runes + sim/client lifecycle** — the store IS the reactive `TriggerLab` object the app binds
  to; it holds the transient snapshot state (voices/log/levels/fps/link/output) and drives the rAF loop;
- **the graph/preset/effect authored model** — `graphs`/`graphNames`/`buses`/`presets`/`effects`/
  `canvasScenes` + graph CRUD/wiring/value-switch (already thin over the pure `store/` slices);
- **the authored-state swap machinery** — `toAuthored`/`applyAuthored`/`applyShow`/
  `resetAuthoredToSeed`/`normalizeGraphs`; it spans EVERY authored cluster (graphs, buses, sections,
  songs, …), so it is the shared centre R23 + R24 both reach through the host, not a slice;
- **the sim play surface** — pad/MIDI/keyboard fire + section recall (out of scope for the split);
- **the WS engine link** — connect/handshake, the `state`/`stats`/`presence`/`input` handlers,
  `engineSync`, `syncShowToServer`/`syncTransport` (drives all five controllers);
- **the S44 system clipboard** — copy/paste closure build + `materializePaste`/remap;
- **routing/project mutators** (Patch graph), **undo/redo**, **autosave + save-status**, and the
  misc UI state (popups, pane sizes, monitor filters, node positions).

A further split would target the graph model or the play/engine surface, but those are the store's
integration core (the swap machinery + WS link couple them) — the god-file's five *separable* domain
clusters are done.
