# R29 тАФ PixLite admin-password for authenticated controllers (GH #108)

## Outcome
Controllers with a non-empty admin password can now authenticate. An admin-password
setting flows **UI тЖТ server (hash) тЖТ project persistence тЖТ PixLite transport**, which
uses it for every management call. An empty/unset password behaves exactly as today.

## What changed (by seam)
- **Transport (`packages/io`)** тАФ no new transport code: `HttpPixliteClient` already
  hashes a `password` (or takes a pre-computed `auth`) into the `?auth=` query param.
  Added the IO-seam **authenticated-path** test (exact hash, distinct from the empty
  default) plus an empty-password-equals-today test.
- **Server (`apps/server`)** тАФ new `controllerMonitor.setAuth(password)`: hashes via
  `authHash` (empty тЖТ cleared), rebuilds the client with the credential, persists
  **only the hash** on `project.controller.auth` (never the plaintext), forgets the
  prior identity/success so a wrong password shows as unreachable, and kicks a poll.
  Wired through the WS handler (`setControllerAuth`, editor-gated by deny-by-default)
  and `main.ts`.
- **Protocol** тАФ new `{ t: 'setControllerAuth'; password: string }` client message
  (plaintext over the local WS; the server is the only hasher, per the LOCKED
  `controller.auth` contract).
- **Web (`apps/web`)** тАФ `store.setControllerAuth`; a masked **Admin password** field
  in `ControllerStatusPanel` (threaded through `OutputStatusPanel` +
  `PatchControllerInspector`). New `password` variant on the `CommitInput` primitive
  (never trims, never round-trips its value, clears after a set). The panel hint
  reflects the device's own `authReqd` + reachability тАФ no new status flag needed
  (a successful poll under `authReqd` *is* proof the password authenticated).

## Root cause / gap found
The `auth` hash already flowed end-to-end (schema тЖТ monitor тЖТ client), but there was
**no way to set it** тАФ `adopt` only carried a hash forward for an already-authed host.
This ticket adds the missing set path; the persistence/transport plumbing was reused.

## Deviations / notes
- `pnpm gates` (the machine-wide wrapper) aborts on this 16-core host with a vitest
  `minThreads/maxThreads must not conflict` error тАФ the gate caps `MAX` workers to 4 while
  vitest's CPU-derived `MIN` exceeds it. This is **pre-existing tooling**, unrelated to R29.
  Verified the exact gate contents (typecheck + full `pnpm -r test`) green by pinning
  `VITEST_MIN_FORKS=1 VITEST_MIN_THREADS=1`. Worth a follow-up fix in `scripts/with-gate-lock.mjs`
  (also set the `MIN` env vars).
- Clearing a set password from the UI isn't exposed (the field only sets non-empty); the
  server `setAuth('')` path supports it and re-adoption resets auth. Low priority.
- Real-hardware probe is the separate manual ticket R28 тАФ not attempted.
