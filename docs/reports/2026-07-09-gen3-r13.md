# R13 — Delay = timeline shift: overlap-based Mix composition (#92)

Branch `gen3r/r13-delay-timeline`. Tests-first, per the spec mandate.

## What changed

Composition membership at a **Mix** is now **temporal overlap at render time**, not
eval-batch membership. When a delayed branch drains into a Mix, it re-composes with the
Mix's **still-live** sibling members (per the Mix blend rules); members whose voices have
**decayed** are absent. `delay 0` stays byte-identical to no delay.

### Core seam (`packages/core`, canonical `TriggerGraph → actions`)
- **`eval-graph.ts`** — `EvalState` gains two optional, engine-owned fields:
  - `mixMemberSnapshots: Map<pad#mixId, MixInputDraft[]>` — the Mix evaluator records its
    members here (merged by origin) as they arrive across eval batches.
  - `isLayerLive(pad, originNodeId)` — origin-keyed liveness.
  On a **drained** batch (`seen.size > 0`), the Mix folds back snapshotted members that are
  not in-batch **and** still live, then sorts by canvas y-order. Immediate/`delay 0`
  behaviour is unchanged; a bare `EvalState` (no snapshots/liveness) stays batch-scoped, so
  the whole existing Mix suite is untouched.
  - `PlayAction`, `PlayDraft`, `MixInputDraft` exported; `PlayAction.originNodeId` added.
- **`types.ts`** — `Voice.pad`, `Voice.originNodeId`, `MixInput.originNodeId`.
- **`voice-pool.ts`** — `spawn` tags each voice with `pad` + `originNodeId` and carries
  `originNodeId` onto mix members; new `VoicePool.isLayerLive(pad, origin)` scans active
  voices (own origin or a mix member's origin).
- **`engine.ts`** — owns the `mixMemberSnapshots` map (cleared on `setShow`); `evalState()`
  exposes it + `isLayerLive`; `applyActions` threads the eval `pad` into `spawn`.

### Offline preview (`apps/web`)
- **`sim.ts`** — mirrors the snapshot map + an `isLayerLive` scan over its own voice list in
  `coreEvalState()`, tags spawned voices with `pad: 'preview'` + `originNodeId`. The sim
  already delegates Gen3 eval to core (R16), so the preview follows the core model directly.

### UI (criterion 4)
- **`ContainerNodeInspector.svelte`** — the Mix inspector now states the canvas y-order
  stacking rule ("Layers stack in canvas order: the topmost branch is the base, and each
  branch below blends over it") above the existing y-sorted layer rows. Reuses the existing
  `.hint` token style (+ `text-wrap: pretty`). Svelte autofixer clean.
- **`shot-seam.ts`** — one sanctioned adapter method `mixWithLayers()` (state op
  `mix-layers`) authors a Mix with two wired branches and selects it.

## Root cause fixed
The delay node defers its subtree to a pending-fire drain that re-enters eval fresh, so a
delayed branch into a Mix only ever saw its own batch: immediate Mix = `[A]`, drained Mix =
`[B]`, never composed. Now the drain re-includes still-live earlier members.

## Deviations & follow-ups
- **Render-time coalescing / frame-exactness = R14.** This slice settles the temporal model
  at the evaluated-actions seam (the spec's canonical Core seam). During overlap the
  re-composed drained Mix and the still-live immediate Mix voice both exist; coalescing that
  double-count (and the "A at frame 31, B at frame 1" render nicety, which needs per-MixInput
  lifetimes) is R14's job, explicitly "defined against your settled temporal model."
- **Adjacent bug (noted, not fixed):** capturing the wired-Mix inspector via `pnpm ui-shot`
  hit an uncaught `validateDrag is not defined` thrown from the xyflow canvas when a Mix node
  with per-row handles renders headlessly. The identifier exists **nowhere** in source, deps,
  build output, or git history; bare-Mix and all baseline shots render fine, and my change
  (a caption line + pure store-API adapter) cannot introduce an xyflow `ReferenceError`. The
  inspector chrome itself renders correctly in ui-shot (Delay inspector captured cleanly), and
  the Mix copy is code- + autofixer-verified. Flagging for triage — likely a pre-existing /
  environmental mix-edge canvas-render issue on the base branch.
