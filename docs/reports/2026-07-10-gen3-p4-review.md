## What was verified

### 1. Store split correctness (R20–R24) — clean
- **Constructor-injected hosts, no sibling reach-ins.** All five controllers
  (`ControllerMonitor`, `MidiController`, `ControllerTest`, `ShowsController`,
  `SectionsController`) take a single injected host interface and never reference a
  sibling controller. Cross-slice dependencies are mediated entirely through the
  store host: `SectionsController` writes the `songs` rune via
  `host.setSongs` → `store.songs` setter → `showsCtl.songs`
  (`sections-controller.svelte.ts:93`), and `ShowsController` re-points the section
  pointer via `host.setActiveSectionId` → `store.activeSectionId` →
  `sectionsCtl` (`store.svelte.ts:523`). `ControllerTest` reads the active pattern
  through `host.currentTestPattern()` off `monitor.status`
  (`store.svelte.ts:481`) rather than duplicating it. Grep for controller→sibling
  references returned nothing.
- **Single source of truth per rune.** No migrated field is still declared as
  `$state` in the store — `songs/songRefs/activeSongId/activeShowId/songLibrary/
  showLibrary` live only on `ShowsController` (`shows-controller.svelte.ts:153–173`);
  `activeSectionId/sectionClipboard` only on `SectionsController`
  (`sections-controller.svelte.ts:59,63`); `status/candidates/scanning` only on
  `ControllerMonitor`. The store exposes them via getter/setter delegators, so the
  public API is byte-compatible.
- **Reactivity across the getter chain holds.** `SectionsController`'s `$derived.by`
  reads the active song through `host.activeSong()` (store getter →
  `showsCtl` derived → `songs` rune); the 25 store.sections + 14 store.shows tests
  exercise this chain and pass.

### 2. Undo-stack integrity across the split — clean, no regression
- The authored snapshot `toAuthored()` reads `this.activeSectionId` and `this.songs`
  **through the delegating accessors** (`store.svelte.ts:1103,1095`), and
  `applyAuthored()` writes them back through the same setters
  (`store.svelte.ts:1134,1117`). So controller-owned state is captured/restored by
  undo exactly where it was before the split.
- Section edits are **not** wrapped in `runUndoable` and the `songs` setter does not
  snapshot — but this is faithful to pre-split behavior: on `main`, `updateActiveSong`
  likewise only did `this.songs = this.songs.map(...)` with no undo push, and `songs`
  was a plain rune. No undo behavior was lost in the extraction.
- `sectionClipboard` is correctly absent from the authored snapshot (transient,
  matches pre-split).

### 3. R19 prune ↔ lint/cache consumers — clean
- Dropped from the public `RenderPlan`: `planNodesById`, `outputId`,
  `incomingFlowEdgesById`, plus `RenderPlanNode`, `RenderPlanNodeCategory`,
  `nodeCategory()`. Repo-wide grep confirms **zero** external consumers of any of
  them (the only `outputId` hit is an unrelated kit-editor field in
  `PatchOutputInspector.svelte:26`).
- `incomingFlowEdgesById` and `outputId` are still **built and consumed internally**
  by `detectUnreachable(...)` (`render-plan.ts:78`) — correctly removed from the
  output surface only, not dead computation.

### 4. Cache invariant (`renderPlanSignature`) after R19 — holds
- R19 did not touch `renderPlanSignature` (structural: kinds + edges). The
  reachability-lint contract comment (`reachability-lint.ts:18`) and `LintCallout`'s
  "compiled UNCACHED, per the render-plan cache contract" note remain consistent;
  render-plan/reachability/scope-lint tests (31) pass.

### 5. Determinism / core purity — clean
- No Node/DOM imports added to the touched core voice files
  (`render-plan.ts`, `reachability-lint.ts`, `scope-lint.ts` are pure).
- R15's legacy Block-tree **evaluator** deletion in `sim.ts` (`evaluate`, `trigger`,
  `switchIndex`, `resolveParams`, `modeWord`) removed only the dead pre-Gen3 path.
  The Block-tree **types** + `treeToGraph` compiler remain live: `fixtures.ts` seeds
  pads as Block trees compiled to `TriggerGraph` via `sim.graph-compilation.ts`. Not
  dead code — correct scope.
- R17 sim→core type alignment carries no `as unknown as` re-introduction in the
