# R05 — Graph lint strip fed by render-plan compile issues (GH #84)

## What changed
A persistent **lint strip** now lives on the trigger-graph surface, rendering the
render-plan compiler's `compileRenderPlan().issues` — which were computed in
`packages/core` but consumed nowhere in the web app. Each row states the problem
plainly and names the next step; the compiler's specific detail (a cycle path) shows
beneath. The strip is **absent** when the graph compiles clean.

- **Surfaced the existing linter, not a second one.** The strip reads
  `voice.compileRenderPlan(selectedGraph).issues` directly — no new validation.
- **Complementary to R03's toasts.** R03 flashes a rejected wire gesture; this is a
  persistent structural surface. It uses the warn family (amber), not the red output
  fault callout — it guides authoring, it doesn't alarm.

## Root cause / context
`compileRenderPlan` emits three issue codes today: `missing-trigger`,
`missing-output`, `flow-cycle`. In the real app a well-formed graph is guaranteed a
trigger + Output anchor (`buildEmptyGraph` + hydrate migration) and cycles are refused
at wire time (`store.connect`), so the live strip is empty for authored graphs — as the
brief anticipated ("render what exists; R06/R07 extend the surface"). The machinery is
wired end-to-end and ready for those codes.
